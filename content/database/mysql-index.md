# 数据库 - MySQL 索引面试题

## 1. 为什么 MySQL 使用 B+ 树作为索引结构？

**答案**：
1. **非叶子节点只存索引**：单个节点能存更多索引，树更矮胖，IO 次数更少
2. **叶子节点存数据**：所有数据都在叶子节点，查询稳定
3. **叶子节点有链表**：范围查询效率高

**对比 B 树**：
- B 树：节点既存索引也存数据，树更高，IO 更多
- B+ 树：非叶子只存索引，树更矮，IO 更少

**记忆要点**：
- B+ 树更矮胖，IO 更少
- 叶子有链表，范围查询快

## 2. 聚簇索引和非聚簇索引的区别？

**答案**：
| 对比项 | 聚簇索引 | 非聚簇索引 |
|--------|----------|------------|
| 数据存储 | 数据和索引在一起 | 数据和索引分离 |
| 叶子节点 | 存完整行数据 | 存主键值 |
| 数量 | 一个表只能有一个 | 可以有多个 |
| 例子 | 主键索引 | 普通索引 |

**记忆要点**：
- 聚簇：索引 + 数据在一起
- 非聚簇：索引存主键，再回表查数据

## 3. 什么是覆盖索引？

**答案**：
查询的列刚好是索引的列，不需要回表查询。

**例子**：
```sql
-- 有索引 (name, age)
SELECT name, age FROM user WHERE name = '张三';  -- 覆盖索引，不回表
SELECT name, age, address FROM user WHERE name = '张三';  -- 需要回表
```

**优点**：
- 避免回表，减少 IO
- 提高查询性能

**记忆要点**：
- 查询列 = 索引列，不用回表

## 4. 索引失效的场景有哪些？

**答案**：
1. **最左前缀原则**：联合索引不满足最左前缀
2. **函数运算**：对索引列使用函数
3. **类型转换**：隐式类型转换
4. **模糊查询**：`LIKE '%xxx'` 左模糊
5. **OR 条件**：OR 两边有非索引列
6. **NOT IN/NOT EXISTS**：负向查询

**例子**：
```sql
-- 失效
SELECT * FROM user WHERE YEAR(create_time) = 2024;  -- 函数
SELECT * FROM user WHERE name = 123;  -- 类型转换（name 是字符串）
SELECT * FROM user WHERE name LIKE '%张%';  -- 左模糊
```

**记忆要点**：
- 最左前缀、函数、类型转换、左模糊、OR、负向

## 5. 如何优化慢查询？

**答案**：
1. **开启慢查询日志**：定位慢 SQL
2. **EXPLAIN 分析**：查看执行计划
3. **优化索引**：
   - 添加合适的索引
   - 避免索引失效
   - 使用覆盖索引
4. **优化 SQL**：
   - 避免 SELECT *
   - 避免子查询
   - 小表驱动大表
5. **优化表结构**：
   - 垂直拆分
   - 水平拆分
   - 分区表

**记忆要点**：
- 开日志 → EXPLAIN → 优索引 → 优 SQL → 优结构

## 6. 事务的 ACID 特性？

**答案**：
| 特性 | 说明 | 实现原理 |
|------|------|----------|
| 原子性（A） | 事务要么全部成功，要么全部失败 | undo log |
| 一致性（C） | 事务前后数据一致 | 最终结果 |
| 隔离性（I） | 事务之间互不干扰 | 锁 + MVCC |
| 持久性（D） | 事务提交后永久生效 | redo log |

**记忆要点**：
- 原子 undo、隔离锁+MVCC、持久 redo

## 7. 事务的隔离级别及解决的问题？

**答案**：
| 隔离级别 | 脏读 | 不可重复读 | 幻读 |
|----------|------|------------|------|
| READ_UNCOMMITTED | ❌ | ❌ | ❌ |
| READ_COMMITTED | ✅ | ❌ | ❌ |
| REPEATABLE_READ（MySQL 默认） | ✅ | ✅ | 部分✅ |
| SERIALIZABLE | ✅ | ✅ | ✅ |

**记忆要点**：
- 读未提交：啥都不解决
- 读已提交：解决脏读
- 可重复读：解决脏读 + 不可重复读
- 串行化：全解决

## 8. MVCC 的实现原理？

**答案**：
MVCC（多版本并发控制）通过以下方式实现：
1. **隐藏字段**：每行记录有事务 ID、回滚指针
2. **undo log**：保存历史版本
3. **read view**：快照读时生成读视图

**工作流程**：
- 事务读取时，根据 read view 判断哪个版本可见
- 如果当前版本不可见，通过回滚指针找历史版本

**记忆要点**：
- 隐藏字段、undo log、read view

## 9. InnoDB 的锁类型？

**答案**：
| 锁类型 | 说明 |
|--------|------|
| 记录锁（Record Lock） | 锁住单条记录 |
| 间隙锁（Gap Lock） | 锁住索引记录之间的间隙 |
| 临键锁（Next-Key Lock） | 记录锁 + 间隙锁 |
| 意向锁（Intention Lock） | 表级锁，表示要加行锁 |

**记忆要点**：
- 记录、间隙、临键、意向
